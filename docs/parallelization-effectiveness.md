# 並列化（`--sub-agents`）はどこで効くか — 効く条件・効かない条件

> このドキュメントは「サブエージェント並列化が実際に速度改善につながるのはどの構成か」を、
> 実運用ログの実測値に基づいて整理したもの。機能の使い方そのものは
> [`sub-agents.md`](./sub-agents.md) を参照。関連: [`local-media.md`](./local-media.md) /
> [`whisper.md`](./whisper.md)。

## 結論（先に要点）

- **並列化は「待ち時間（I/O）を重ねて省く」技術。** ボトルネックが「待ち」ではなく
  「計算（CPU/GPU）」のときは、重ねる余地がないので効かない。
- **ローカル動画 + MLX Whisper（Apple Silicon 単一 GPU）構成では、サブエージェント並列化は
  ほぼ無効**。壁時計時間の 85〜90% を占める文字起こしが、単一 GPU の計算バウンド処理であり、
  かつグローバルロックで直列化されているため。
- 並列化が効くのは **YouTube 経由 ×（Whisper 不要な）字幕ありの I/O バウンド構成**、または
  **複数 GPU／能動冷却のある機材**。
- Stage 05 の失敗（α バッチの JSON 崩れ）は **LLM 出力信頼性という別軸の外部要因**であり、
  並列化の有無とは無関係。

## 実測：時間はどこに消えるか

ローカル 16 本（`--local-media`, MLX `large-v3-turbo`）の本番ランでの、動画あたりの
処理時間（ほぼ Whisper 文字起こし時間）:

```
23:10, 11:26, 20:38, 11:10, 20:07, 14:49, 11:40, 17:05,
15:15, 11:49, 11:04, 26:44, 19:46, 10:56, 08:24, 06:54
合計 ≈ 約4時間
```

同じランの他工程:

| 工程 | 1 本あたりの壁時計時間 | 16 本合計（目安） |
|---|---|---|
| 01 文字起こし（Whisper / MLX） | 7〜27 分 | **≈ 4 時間（全体の 85〜90%）** |
| 02 要約（LLM, haiku） | 数十秒〜数分 | 〜30 分 |
| 03 キャプチャ（ffmpeg） | 数秒〜1 分 | 数分 |
| 04 学習（LLM, sonnet） | 数十秒〜数分 | （02 と合算で〜30 分） |
| 05 統合（Agent Teams） | — | ≈ 12 分（701.9s） |

→ **支配項は完全に 01 の Whisper**。ここを縮めない限り全体は縮まない。

## なぜローカル + MLX では並列化が効かないか

### 原理：並列化は「待ち」を重ねる技術

| ボトルネックの種類 | 例 | 並列化の効果 |
|---|---|---|
| **I/O バウンド**（待ちが支配的） | YouTube ダウンロード、字幕 API、ネットワーク | ◎ 待ち時間を重ねられるので効く |
| **計算バウンド**（CPU/GPU が支配的） | MLX Whisper の推論 | ✗ 重ねる「待ち」がない／単一資源を取り合うだけ |

オフライン + MLX にした瞬間、ボトルネックが「ダウンロード待ち（I/O）」から
「GPU 推論（計算）」へ移る。これが効かなくなる根本理由。

### 物理的な理由（MLX × M2 Air）

- MLX Whisper は **M2 Air の単一 GPU** を使い切る。サブエージェントを 3 つ立てても
  GPU は 1 個しかないので、3 プロセスが同じ GPU を時分割で取り合うだけ。総 GPU 秒数は
  変わらず速くならない。
- Whisper は**グローバルロック／セマフォで直列化**されている（同時に走らせない設計）。
- MLX モデルは 1 本あたり 2〜3GB。同時起動はメモリを圧迫し、12GB 以下に収めたい
  M2 Air 24GB では**他作業がクラッシュ**しうる。
- ファンレス筐体のため、複数同時実行は**サーマルスロットリング**でむしろ遅く・不安定になる。

### 上限の試算

LLM 段（02/04）は理屈上は本数ぶん並列にできるが、全体の 1 割程度。仮に完璧に並列化しても
**約 4.5 時間 → 約 4 時間**になる程度で、メモリ／熱のリスクに見合わない。

## Stage 05 の失敗は並列化と無関係

本番ランで観測された

```
α batch 1/2 failed: no JSON object found in response
chapters: 0
```

は、**LLM が構造化 JSON を途中で崩した出力信頼性の問題（外部要因）**。並列化していても
いなくても発生し、並列化では防げも直せもしない。対処は「より強いモデル（opus 等）で
`--synthesis-only` を再実行」（[`sub-agents.md`](./sub-agents.md) の `--synthesis-only` 参照）。
SDK 版では config の `models.alpha` / `models.beta` / `models.leader` を信頼性の高いモデル
（または別プロバイダ）に切り替えるのも有効。

## では並列化はいつ使うのか

機能自体は正しく、**I/O バウンドまたは複数 GPU 向け**。

| 構成 | ボトルネック | 並列化の判断 |
|---|---|---|
| YouTube 経由 × 字幕あり（Whisper 不要） | ダウンロード／字幕取得の I/O 待ち | **使う**（シャードで待ちを重ねて短縮） |
| YouTube 経由 × Whisper フォールバック多発 | GPU 推論 + YouTube レート制限 | 慎重に。並列度を上げると IP ブロックを誘発しやすい |
| ローカル動画 × MLX（単一 GPU） | 単一 GPU の計算 | **使わない**（効果なし・メモリ／熱で逆効果） |
| 複数 GPU／能動冷却のある機材 | GPU 推論（並列可能） | **使う**（Whisper 自体を真に並列実行できる） |

## 推奨

### ローカル + MLX（M2 Air など単一 GPU）

`--sub-agents` を付けず、**単一プロセス（`--concurrency 1`）のまま**が正解。理由:

1. ボトルネックが単一 GPU の計算で、並列化の余地がない
2. 並列化はメモリ超過・サーマルスロットリングで逆効果
3. 実運用で 16/16 成功しており動作は健全
4. 唯一の失敗（Stage 05）は並列化と無関係な外部要因

速くしたい場合の本当のレバーは並列化ではなく:

- Whisper モデルを軽くする（`large-v3-turbo` → `medium`/`small`、精度とのトレードオフ。
  [`whisper.md`](./whisper.md) 参照）
- オフライン処理なので「夜間にバッチで回しっぱなし」と割り切る

### YouTube 経由・字幕あり、または複数 GPU 機材

このときは `--sub-agents N` が本来の効果を発揮する。使い方は [`sub-agents.md`](./sub-agents.md)。

## まとめ（一文）

> **並列化は「待ち」を重ねる技術。いまのローカル + MLX のボトルネックは「待ち」ではなく
> 単一 GPU の「計算」だから、重ねる余地がなく効かない。** 並列化が活きるのは I/O バウンド
> または複数 GPU の構成。
